# Agent 10 Prompt: User-Incremental Governance And Working Method

## First Read These Files

Read before any edit:

1. `RULES.md`
2. `WIP_GUIDELINES.md`
3. `CONTRIBUTING.md`
4. `README.md`
5. `STATUS.md`

## Mission

Strengthen the governance language so the repo consistently enforces a user-incremental working model instead of allowing long stretches of invisible internal work to appear as “progress.”

## Why This Matters

The last conversations made this explicit:

- the backend exists
- the product still feels invisible
- some docs sounded more complete than the user experience actually is
- future work should land in small, testable, user-facing increments whenever possible

This task should turn that lesson into durable repository guidance.

## Your Ownership

You own only:

- `RULES.md`
- `WIP_GUIDELINES.md`
- `CONTRIBUTING.md`
- `README.md`

Do not edit:

- implementation code
- SDK files
- UI files
- `PLAN.md`
- `BITACORA.md`

## Concrete Deliverable

Improve the working-method language so it is explicit that:

- each round should preferably leave one more thing runnable, visible, or inspectable
- “done” should not be used to imply finished product experience when only an internal slice is complete
- parallel work should be divided by disjoint write ownership
- documentation must state clearly whether a change is user-visible or internal-only

## User-Increment Requirement

The governance changes should make future work more honest, more testable, and less confusing for users and collaborators.

## Collision Avoidance

Do not touch implementation or planning files. Stay in governance/contribution docs only.

## Completion Standard

You are done when:

- the repo’s working-method docs clearly enforce user-incremental delivery
- contributor expectations are more explicit
- the docs reduce the chance of future confusion about what is actually usable
